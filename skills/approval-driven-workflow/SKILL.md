# 承認駆動ワークフロースキル

このスキルは、変更を伴う作業を段階的に進めるための共通ワークフローを提供します。

## 有効化するタイミング

- ユーザーが実装、修正、設計変更、リファクタリング、または変更を前提とした調査を依頼した場合
- 作業前に計画を提示し、承認を得てから進める必要がある場合
- 実作業の後に、skill として取り込むべき知見や手順の有無を確認したい場合
- コード変更と skill 変更を別々の承認ゲートで管理したい場合

## 適用しないタイミング

- 雑談、一般的な質問、単純な説明、翻訳、要約など、変更を伴わない依頼の場合
- 単なる読解、説明、現状把握のみを目的とした依頼の場合
- ユーザーが明示的に即時実装を求めており、計画提示や承認待ちを不要とした場合
- 既存の作業フローよりも、緊急対応や短い一問一答を優先するべき場合

## 基本方針

- 依頼を受けたら即座に編集や変更を始めるのではなく、まず計画を立てる
- 計画の承認前は、読取り、調査、影響範囲の把握に留める
- コーディングに対する変更を加える場合は、実装や修正の前に必ず読取りと影響範囲の確認から開始する
- 実作業は、承認後に適切な作業スキルを選択して進める
- skill の追加・更新は、通常の作業とは別の承認を得てから行う

## 実行フロー

### 計画作成

- 計画系のスキルを用いて、作業の目的、前提、調査事項、実施手順、検証方針を整理する
- 影響範囲、想定リスク、未確定事項を明確にする
- 必要に応じて、実作業で使う候補スキルを整理する

### 計画承認

- 計画をユーザーに提示し、承認を得る
- 承認は、ユーザーによる明示的な同意がある場合にのみ成立する
- 条件付き承認の場合は、その条件を計画に反映してから進める
- 無応答は承認とみなさない
- 計画が承認されるまでは、コード編集、設定変更、skill 変更を行わない
- 承認時に条件付きの指示や優先順位が追加された場合は、以後の作業に反映する

### 実作業

- 承認された計画に従って、利用可能な作業スキルを選択して進める
- 該当する作業スキルが存在しない場合は、本スキルの方針に従って読取り、実装、検証、レビューを代替的に進める
- コーディングに対する変更を加える場合は、読取り系のスキルで既存実装、関連ファイル、影響範囲を確認してから実装に入る
- 実装や修正の後は、最終段階としてレビュー系のスキルで変更内容、リスク、見落とし、検証不足を確認する
- 実装や修正の後は、ドキュメント同期系のスキルで関連ドキュメントの更新要否を確認し、必要な更新を行う
- 計画から外れる追加作業が必要になった場合は、影響を整理して必要に応じて再確認する

### skill 改善提案

- 実作業を通じて再利用価値の高い知見が得られた場合は、それを整理する
- 新しい skill として切り出すべき手順があるか、既存 skill に不足、重複、曖昧さ、更新漏れがないかを確認する
- 提案する場合は、追加候補、更新候補、変更理由、期待される効果を含めて提示する

### skill 変更承認

- skill の追加・更新提案をユーザーに提示し、承認を得る
- 承認は、ユーザーによる明示的な同意がある場合にのみ成立する
- 条件付き承認の場合は、その条件を変更内容に反映してから進める
- 無応答は承認とみなさない
- この承認は、通常の作業成果物に対する承認とは分けて扱う
- 承認されていない skill 変更は行わない

### skill 変更

- 承認された内容に従って、`~/.agents/skills` 配下の skill を追加または更新する
- 既存 skill と命名、説明、構成、粒度の一貫性を保つ
- 変更後は、今回の知見が将来の作業で再利用しやすい形になっているかを確認する

## skill 提案時の観点

- 複数回出現しそうな手順か
- 作業の品質や一貫性に直接効く内容か
- 単なる今回限りのメモではなく、skill として独立できるか
- 既存 skill の追記で足りるか、新規 skill の方が適切か
- 他の skill と責務が重複していないか
